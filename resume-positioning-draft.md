# 个人介绍文档

## 自然叙事版

我想做的是软件产品相关的事情，但不只是写代码。

我更感兴趣的是：一个真实需求是怎么出现的，为什么有人愿意为它付费，技术应该怎么介入，产品怎么设计，方案怎么讲清楚，最后怎么让它真正被使用、被验证、产生价值。

所以我对自己的定位，不是单纯的 Developer。  
我更接近一个产品型、方案型的技术人：懂技术实现，也愿意理解用户、业务、商业和交付。

我喜欢从整体看问题。  
遇到一个需求，我不会只想“这个功能怎么做”，我会先想它是不是一个真问题，用户为什么需要它，现在的方案为什么不够好，做出来之后谁会用，怎么判断它有没有价值。

我相信技术本身很重要，但技术不应该孤立存在。  
技术要进入真实场景，要解决具体问题，要能被别人理解，也要能带来结果。

我希望自己未来能做产品经理、解决方案工程师、技术售前、销售工程师这一类工作，也希望最终成为创业型技术人。  
这些方向看起来不同，但底层能力其实很接近：理解需求，拆解问题，表达方案，判断价值，推动事情往前走。

别人对我的评价里，比较常出现的是有领导感、思维比较成熟、比较靠谱，遇到突发问题时能稳住场面。  
我不一定是反应最快的人，但我会尽量把事情想清楚，把结构理顺，把该负责的部分接住。

我比较反感只抱怨、不解决问题的人。  
也反感遇到困难就撂挑子、把事情当成别人的事的人。  
我觉得一个人能不能长期合作，不只看能力，也要看他是不是愿意负责、能不能守住基本的诚信和边界。

我看问题的方式比较偏求真和结果。  
我不太喜欢只讲立场、情绪或者漂亮话。一个判断对不对，最后还是要看证据、反馈和结果。  
如果新的事实证明原来的想法不对，那就更新判断。  
我相信概率，相信动态调整，也相信一个人真正可靠的地方，不是永远不犯错，而是能不能持续接近真实。

## README 短版

我不只把自己定义为技术人员。

我想做的是软件产品、解决方案和商业价值之间的连接：理解需求，设计方案，讲清技术，推动落地。

我的方向更接近 Product-minded Solution Engineer。  
技术是基础，但我更关心技术进入真实场景之后，能不能解决问题、建立信任、产生结果。

我相信求真、看本质、讲证据，也接受判断需要不断更新。  
对我来说，好的方案不是听起来高级，而是能被理解、能被执行、能被验证。

## 简历摘要版

职业方向偏软件产品、解决方案与商业技术结合，关注如何把真实需求转化为可理解、可落地、可验证的产品或方案。

具备技术学习与实现基础，同时重视产品思维、业务理解、方案表达和结果判断。长期方向是 Product-minded Solution Engineer，并向产品经理、解决方案工程师、技术售前、销售工程师及创业型技术人方向延展。

思维方式上重视事实、证据、本质和结果，习惯根据反馈动态更新判断。做事强调责任感、结构化拆解和持续推进，反感空抱怨、撂挑子和不把事情当自己事的工作方式。

## 写进主页时的原则

GitHub 主页主要还是放技术、工具、项目、数据和博客入口。  
产品思维、商业判断、职业观和价值观不要在主页上大段展开，只保留一点影子。

合适的方式是：

- 首屏用少量词表达方向，比如 Product-minded、Solution Engineer、软件产品、解决方案。
- 技术栈和 GitHub 数据继续保留，作为工程基础。
- 写作区放博客入口，让别人看到我不仅写技术，也写思考。
- 五个维度不直接作为主页大标题，但可以通过文章标题露出来。
- README 的整体气质应该是技术主页，不是价值观宣言。

## 博客五篇文章规划

这五篇文章可以放在博客上，作为更深一层的个人表达。  
GitHub 主页只需要通过“最新文章”或“写作与思考”露出它们，不需要专门解释。

### 1. 行业能力：一个产品为什么要先进入真实场景

重点不是写“行业能力”这个词，而是写：  
为什么做软件不能只看功能，还要看谁使用、谁决策、谁买单、谁承担成本、谁获得收益。

可以展开：

- 需求为什么不能脱离场景
- 行业规则如何影响产品设计
- 客户、用户、决策者、付费者可能不是同一个人
- 技术方案进入真实世界后会遇到哪些约束

### 2. 专业能力：技术不是终点，但没有技术就没有落地

重点是说明技术的真实位置。  
技术不是用来炫耀的，但它是把想法做出来、把方案验证出来的基础。

可以展开：

- 为什么硬技能仍然重要
- Demo、原型、自动化、数据展示为什么能降低沟通成本
- 前端、工程化、AI 工具、数据分析这些能力如何服务产品
- 技术能力如何支撑方案判断

### 3. 职业能力：靠谱的人会降低协作成本

重点写责任心、执行力和稳定反馈。  
不要写得像鸡汤，可以写真实工作里最怕什么：不确认目标、出问题不说、快到截止才暴雷、撂挑子。

可以展开：

- 接到任务为什么要先确认目标
- 为什么及时反馈比闷头干更重要
- 遇到突发问题时怎么稳住事情
- 为什么“把事情当自己的事”很重要

### 4. 认知能力：用证据、反馈和概率更新判断

重点写贝叶斯式的思维方式，但不要写成哲学论文。  
可以从日常判断、产品判断、技术方案判断切入。

可以展开：

- 为什么很多判断都不是非黑即白
- 为什么初始判断只是一种概率
- 新证据出现时为什么要更新观点
- 求真、本质、科学、辩证、结果导向如何影响做事方式

### 5. 人品与底线：长期合作最怕的是不可信

重点写诚信、底线和长期主义。  
这篇文章可以很有力量，但要克制，不要说教。

可以展开：

- 为什么能力强但不可信的人协作成本很高
- 不装懂、不甩锅、不夸大为什么重要
- 为什么短期好看不值得破坏长期信任
- 为什么人品不是抽象道德，而是合作效率
